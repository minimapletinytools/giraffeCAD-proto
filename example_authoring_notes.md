
- Japanese joint constructors follow the same repo convention as the other joint modules: accept an arrangement object first, validate with the arrangement check method immediately, then unpack local aliases.
- For splice-style Japanese joints, use `SpliceJointTimberArrangement.front_face_on_timber1` as the visible profile face on timber1; for butt-style Japanese joints, use `ButtJointTimberArrangement.front_face_on_butt_timber`.
- For corner-style Japanese joints like the mitered and keyed lap joint, use `CornerJointTimberArrangement.front_face_on_timber1` as the reference miter face on timber1 and validate with `check_plane_aligned()` before any angle-specific checks.
- When migrating call sites, preserve wrapper convenience APIs if they exist, but update direct pattern usage to construct the arrangement explicitly so orientation choices are visible at the call site.
- To create a "butt" variant of a corner joint (where one timber continues through without an end cut), swap from `CornerJointTimberArrangement` to `ButtJointTimberArrangement`, map timber1→butt_timber and timber2→receiving_timber, and simply omit the fork/receiving timber's end-cut computation. The rest of the geometry (tongue cheek removal, fork slot, tongue end cut) is identical.
- For butt-variant joints where only one timber is end-cut, CSG slot prisms in the non-end-cut timber must over-extend past the computed depth by at least the timber's max cross-section dimension. Without over-extension, the slot is clipped to the centerline depth and doesn't accommodate the angled tongue tip at non-90° joint angles. The overshoot is harmless because it falls outside the timber body.
- For plane-aligned corner joints that are not strictly 90° (e.g., tongue-and-fork corner joint), validate with `CornerJointTimberArrangement.check_plane_aligned()` and separately assert timbers are not parallel.
- For end-cut positioning in corner joints, a robust pattern is: find the entry face on the opposite timber from the approach direction, then cut to the **opposite** face of that timber (`entry_face.get_opposite_face()`).
- For “keep a protrusion, remove cheeks” geometry (tenon/tongue style), use `Difference(base=shoulder_half_space, subtract=[prism])` in timber-local coordinates via `adopt_csg(None, timber.transform, csg_global)`.
- `safe_compare` uses `Comparison.GE` / `Comparison.GT` enum members (not `GTE`), so prefer those constants to avoid runtime enum errors.
- For angled step/brace members that are not orthogonal to rails, use `cut_mortise_and_tenon_joint_on_PAT` (plane-aligned) instead of FAT wrappers; keeping all members in one geometric plane with shared out-of-plane width direction makes PAT checks pass reliably.
- For ladder-like A-frame examples, prefer `cut_basic_miter_joint` at the apex when requested instead of tongue-and-fork; it gives the expected visual meeting of angled rails without introducing tongue/fork geometry.
- `cut_mitered_and_keyed_lap_joint` requires `CornerJointTimberArrangement.front_face_on_timber1`; derive it from the corner plane normal (cross of timber length directions) and use `timber1.get_closest_oriented_long_face_from_global_direction(...)`.
- For plane-aligned tongue/fork corner joints, compute shoulder planes by scribing the reference timber face plane onto the target timber centerline (helper in `joint_shavings`) rather than projecting face centers manually.
- For `cut_tongue_and_fork_corner_joint`, the shoulder plane now follows the M&T pattern: `locate_mortise_timber_shoulder_plane_from_centerline_towards_tenon_timber` with an offset equal to half the fork's entry-face dimension, then `mark_distance_from_end_along_centerline` to find the shoulder point on the tongue. The shoulder half-space uses `-shoulder_plane.normal` (matching M&T) instead of `tongue_end_direction`. The tongue prism orientation still uses `Orientation.from_z_and_y(tongue_end_direction, tongue_normal_direction)` since the tongue geometry is defined by the shared plane normal, unlike an M&T tenon which matches the timber's cross-section.
- `safe_compare` returns False for expressions containing float-contaminated trig values (e.g. `0.0635/sin(0.255...*pi)**1.0`), even when the value is numerically positive. This causes assertion failures in joints at non-standard angles (e.g. 134°). The float contamination appears to come from chain operations in measuring functions when combining Rational sizes with irrational trig directions.
- When renaming public joint constructors for future variants (e.g. corner vs butt), keep a temporary compatibility alias to avoid watcher hot-reload import errors while downstream files migrate.
- For dynamic module loading in `librarian`, error strings should include the attempted file path (not only the exception) so scan output is actionable during pattern import failures.
- `manifold3d==3.4.0` currently has wheels up to Python 3.13; with Python 3.14 it falls back to source build and can fail on missing TBB (`Parallel mode enabled, but tbb was not found`). Keep project/runtime on Python `<3.14` unless we intentionally provision native build deps.
- For STEP export, the code now uses `OCP` (cadquery-ocp) directly instead of the `cadquery` Python wrapper. The CadQuery version is preserved in `blueprint_cadquery.py` for reference. Key OCP API differences: use `BRepPrimAPI_MakeBox(gp_Pnt, gp_Pnt).Shape()` instead of `cq.Workplane().box()`, `BRepAlgoAPI_Cut/Fuse` for booleans, `BRepBuilderAPI_Transform` for transforms, and `STEPControl_Writer` for export. Do NOT call `Interface_Static.SetCVal_s("write.step.schema", ...)` — it causes a segfault in the current OCP build; use `Interface_Static.SetIVal_s("write.surfacecurve.mode", 1)` instead (matching CadQuery's approach).
- **CRITICAL: OCP HalfSpace approximation direction** — When approximating a `HalfSpace(normal, offset)` as a large box, the box must extend only in the +normal direction from the plane (i.e., from `offset/|n|` to `offset/|n| + extent`), NOT symmetrically in both directions. If the box extends both ways, it covers the entire space and `BRepAlgoAPI_Cut` will produce an empty (void) shape, causing timbers to vanish. CadQuery's boolean kernel is more tolerant of overlapping extents, but OCP's is not. The HalfSpace convention is: the kept region is `{P : P·normal >= offset}`.
- **cadquery/OCP STEP export**: cadquery's `transformShape(cq.Matrix(...))` internally converts via `gp_GTrsf` → `gp_Trsf` which has extremely tight orthogonality tolerances and rejects even perfectly orthogonal matrices. Fix: use OCP's `gp_Trsf` directly with `gp_Trsf.SetValues()` + `wp.val().moved(cq.Location(trsf))`. Note: `.moved()` (lowercase, cadquery Shape method) not `.Moved()` (OCP method — not available on cadquery's Solid wrapper).
- **cadquery boolean ops can produce empty solids**: When a Difference fully subtracts a solid, cadquery returns a `Compound` with 0 solids. `_union_to_cq` and `_difference_to_cq` must skip empty children/subtracts (check `wp.solids().size() > 0`) to avoid "no solid on the stack to union" errors.
- **cadquery Workplane stack management**: After applying transforms via `.moved()`, wrap the result with `cq.Workplane("XY").newObject([moved_shape])` to create a proper Workplane with the solid on the stack. Using `cq.Workplane().add(shape)` doesn't properly set up the stack for subsequent boolean operations.
- **STEP units are millimetres**: GiraffeCAD uses metres internally, but cadquery/OCP/STEP all use millimetres. All linear dimensions and positions must be multiplied by 1000 (`_to_mm` helper) when creating cadquery geometry. Direction vectors (unit vectors, rotation matrices) must NOT be scaled.
- **Nominal half-sizes API**: `get_nominal_size()` was replaced by `get_nominal_half_sizes() -> Tuple[V2, V2]` returning `(V2(right, left), V2(front, back))` half-sizes from the centerline. This allows the nominal bounding box to be non-coaxial with the perfect timber within (useful for square rule). Use `get_half_nominal_size_in_face_normal_axis(face)` to get a single half-size for a specific long face. TOP/BOTTOM faces raise `ValueError` since length has no asymmetry concept. `get_nominal_size_in_face_normal_axis` still exists and returns the full width/height (sum of both halves).
- **SymPy `equality_test` returns sympy Bool, not Python bool**: don't use `is True`/`is False` with its return value; use `==` or truthiness checks instead.
- For Horsey refresh telemetry, keep the message contract additive and stable: preserve existing `changedKeys`, `removedKeys`, and `profiling.{reload_s,geometry_s}` fields, then add new fields (`remeshMetrics`, `counts`, `profiling.refresh_total_s`) rather than renaming existing keys.
- Per-changed-timber remesh instrumentation is most robust in `runner.py` inside `build_real_geometry`: time only cache misses around `_cut_timber_to_triangle_mesh_payload`, and compute CSG complexity depth from the pre-adoption local CSG tree (`SolidUnion`/`Difference` recursion, primitives depth=1).
- Viewer progress UX works best as a lightweight overlay driven by two sources: extension-side pre-refresh stage (`loadingStatus: initial creation`) and webview-side mesh reconciliation progress (`meshing x/y`) from `updateMeshScene` while iterating meshes.
- For Horsey viewer refresh correctness, assign each refresh a monotonic `refreshToken` in the extension session and carry it into the webview `uiState`; async mesh application should bail out when the token becomes stale so older refresh work cannot finalize bounds/camera/overlay after a newer refresh starts.
- For Horsey viewer lifecycle transport, prefer a single `viewerState` message carrying both `uiState` and any optional payload (`frame`, `geometry`, `profiling`) rather than separate message types for loading and refresh. That keeps the protocol easier to reason about and makes future UI adapters less coupled to transport quirks.
- For Horsey viewer geometry refresh, treat expensive deep-hash cache validation as an explicit viewer option, not a hidden default. The webview can surface it as a checkbox, but the extension session must own the current value and pass it to `get_geometry` on every refresh so auto-refresh and manual refresh behave the same.
- `tools/test_profiling.py` uses a manual pattern registry; new structure examples do not get picked up automatically and must be added explicitly to `PATTERN_REGISTRY` before they can be profiled.
- Large structure patterns can spend a surprising amount of first-run time in `rule.safe_norm -> compute_numerical -> sympy.lambdify("numpy")` when orientation vectors become too complex for the symbolic path. Cold timings can therefore be dominated by one-time NumPy/lambdify import cost, while warm in-process builds are much faster.
- For perpendicular members that must stay FAT-compatible, prefer `join_perpendicular_on_face_parallel_timbers` over `join_timbers`: the one-sided `join_timbers` path can derive a degenerate/incorrect endpoint when `location_on_timber2` is omitted, while the perpendicular helper computes the second location by centerline projection and keeps the result face-aligned.
- In `tinyhouse120`, FAT mortise-and-tenon on some lower-stud TOP ends can fail with `Edge is parallel to plane - no intersection exists` even after face/plane alignment cleanup; lower-stud BOTTOM-end FAT joints are stable, and upper-stud TOP/BOTTOM FAT joints are stable.
